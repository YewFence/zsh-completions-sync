# 核心概念

`zcs` 的核心流程很简单。它先加载注册表，再按注册表找到每个工具的补全来源，最后把生成结果写成 zsh 能识别的 `_tool` 文件。

## 注册表

注册表是一个 TOML 配置，用来描述哪些工具需要生成补全脚本，以及这些补全脚本从哪里来。

注册表分为三层，优先级从低到高依次是内置注册表、用户配置、项目配置。后加载的配置会覆盖前面同名工具的字段，所以你可以在用户配置里禁用内置工具，也可以在项目配置里调整某个工具的项目补全行为。

## 全局补全

全局补全适合稳定安装在系统或用户环境里的工具，例如 `mise`、`gh`、`docker`、`starship`。

运行下面的命令会生成全局补全。

```zsh
zcs generate
```

默认输出目录是 `~/.zsh/completions`。`zcs init global` 会把这个目录加入 `fpath`，然后调用 `compinit` 让 zsh 加载补全。

## 项目补全

项目补全适合跟项目工具链绑定的命令，例如当前项目里的 `pnpm`、`uv`、`bun`、`kubectl`。

运行下面的命令会生成项目补全。

```zsh
zcs generate --scope project
```

默认输出目录是当前目录下的 `.completions/zsh`。`zcs init project` 会生成项目补全并把该目录加入 `fpath`。

项目补全目录应该排在全局补全目录之前，这样当前项目的新 shell 会优先使用项目内工具版本生成的补全。

## 补全来源

每个工具至少需要声明一个补全来源。`zcs` 支持从命令标准输出读取，从本地文件读取，从远程 URL 读取，也支持从 Git 仓库读取单个文件。

当工具声明了 `pre-command` 时，`zcs` 会先运行预处理命令，再读取补全来源。这适合那些只能先把补全脚本写到文件的工具。

## 工具检查

默认情况下，`zcs` 会检查 `PATH` 里是否存在与工具同名的可执行文件。工具不存在时会跳过生成，工具存在但生成失败时会输出警告，并保留已有补全文件。

如果工具名和可执行文件名不一致，可以用 `check` 自定义检查方式。如果不想检查工具是否存在，也可以把 `check` 设置为 `false`。
